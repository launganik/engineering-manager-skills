**Section 4: Calendar Signals** (only when sources.calendar = true)

**Section 5: Two-Signal Rule**

```
IMPORTANT - Flag Conservatively:

A person should only appear as flagged (yellow or red) if:
  (a) Two or more signals are below their individual thresholds, OR
  (b) One signal exceeds 2 standard deviations from their personal baseline

Do NOT flag a person based on:
  - One signal that is slightly below baseline (within 1 stddev)
  - Team-relative comparison (e.g., "lowest PRs on the team")
  - Absolute counts without baseline comparison (except for the explicit absolute thresholds listed above)

When in doubt, do not flag. A false negative (missed flag) is preferable to a false positive (unfair flag) in every case.
```

**File 1: BASELINES.md**

This file provides inline computation instructions for Claude to calculate rolling 8-week baselines without external tools. It must be self-contained and executable.

Structure:

```markdown
# Team Health - Baseline Computation Reference

Loaded by: /team-health:pulse (read and follow before computing any baselines)
Purpose: Instructions for computing and updating 8-week rolling baselines inline.

## What a Baseline Is

For each person × metric pair, the baseline is:
- A sliding window of at most 8 weekly observations
- Precomputed mean and standard deviation stored in baselines.json
- A deviation threshold: current value vs mean ± N*stddev determines flag status

Baselines are personal - Alice's baseline is derived from Alice's history only, never the team average.
```

Then include these sections with full worked examples:

**Section: Reading a Baseline**

Instructions: Read baselines.json. Find the person by slug. For each metric, the window array is the last N weekly values (oldest first). mean and stddev are precomputed. last_value is the most recent value.

**Section: Updating a Baseline (after a pulse run)**

Step-by-step arithmetic instructions:

1. Fetch the current week's value for each metric from the MCP source.
2. Append the new value to the window array.
3. If the window length exceeds baseline_window_weeks (default 8), remove the first (oldest) element.
4. Recompute mean: sum all values in window / count of values.
5. Recompute stddev: sqrt( sum( (x - mean)^2 for x in window ) / count ). If only 1 value in window, stddev = 0.
6. Set last_value = the new value just added.
7. Update computed_to = today's date.
8. Write the updated baselines.json.

Include a worked example with real numbers:
- Window before: [2, 3, 1, 2, 2, 3, 2, 2] (8 values)
- New value: 1
- Window after: [3, 1, 2, 2, 3, 2, 2, 1] (oldest dropped, new appended)
- Mean: (3+1+2+2+3+2+2+1)/8 = 16/8 = 2.0
- Stddev: sqrt( ((3-2)^2 + (1-2)^2 + (2-2)^2 + (2-2)^2 + (3-2)^2 + (2-2)^2 + (2-2)^2 + (1-2)^2) / 8 ) = sqrt(4/8) = sqrt(0.5) ≈ 0.71

**Section: Comparing to Baseline**

Instructions for flag determination:

- Compute: deviation = (current_value - mean) / stddev (if stddev == 0, deviation = 0 unless current != mean)
- Flag if deviation < -2.0 (for metrics where low = bad: PR count, commit days, review count)
- Flag if deviation > +2.0 (for metrics where high = bad: review lag, response latency)
- For absolute-threshold signals (meeting load, stuck tickets), use the threshold directly, no baseline comparison needed

**Section: First-Run Baseline (no history)**

When a person has no entries in baselines.json yet:
- Do not flag them. No baseline = no comparison possible.
- After the first pulse run, write their current week's values as the initial window.
- After 3 or more weeks of data, baselines become meaningful. Before that, surface raw values only.

**Section: Provenance Fields**

When writing baselines.json, always include in each person's entry:
- computed_from: earliest date in the window (ISO date)
- computed_to: today (ISO date)
- source: array of MCP source names that contributed data (e.g. ["github", "jira"])
- window_weeks: the window size used (from config.json baseline_window_weeks, default 8)

**File 2: PRIVACY.md**

This file encodes mandatory output language rules. Every command that generates output visible to the manager must load and follow this file.

Structure:

```markdown
# Team Health - Output Language Rules

Loaded by: ALL commands that generate output (/team-health:pulse, /team-health:prep, /team-health:skip-level, /team-health:retro-prep)
Purpose: Mandatory language rules. These rules protect your direct reports from being described in ways that are inaccurate, unfair, or harmful to their careers.

## The Core Principle

Team Health surfaces behavioral observations, not interpretations. The manager decides what to do with them. This tool does not diagnose, score, or judge people.
```

Then include these four rule sections, each with approved and prohibited examples:

**Rule 1: Signals, Not Diagnoses**

Prohibited: "Alex seems burned out", "Jordan is disengaged", "Sam appears to be struggling"
Approved: "Alex's PR review participation is down 40% vs. 8-week baseline", "Jordan's commit activity dropped from 4 days/week to 1 day/week", "Sam has 2 tickets blocked for >1 sprint"

Why: Diagnoses are irreversible labels. Behavioral observations are facts that can be discussed and corrected.

**Rule 2: No DM Content**

Prohibited: Any reference to private message content, direct message sentiment, private conversation tone
Approved: Public channel participation counts, @-mention response latency in public channels only

Why: Accessing DM content without consent is a privacy violation and destroys manager-IC trust if discovered.

**Rule 3: No Team-Relative Scoring**

Prohibited: "Alex is the least active committer on the team", "Jordan is in the bottom quartile for PR reviews", "Team average is X, Sam is below average"
Approved: "Alex's commit days dropped from their personal 8-week average of 4.1 to 1 this week", "Jordan's PR review count is below their personal baseline"

Why: Team-relative scoring creates surveillance culture and penalizes people in high-performing teams for being "merely good."

**Rule 4: No Psychological Labels**

Prohibited: "Alice may be experiencing burnout", "Bob's behavior suggests low morale", "Carol seems anxious about the project"
Approved: "Alice's Slack participation dropped significantly this week", "Bob has not merged any PRs in 2 weeks (personal baseline: 2.1/week)", "Carol has 3 blocked tickets open"

Why: Psychological labeling is outside an EM's scope, creates legal risk, and is usually wrong.

**Section: Required Disclaimer**

Every pulse and prep output must include this disclaimer:

> These signals are indicators, not diagnoses. Always talk to your people before drawing conclusions.

Place the disclaimer at the end of the output, after the main content. Do not bury it.

**Section: Graceful Degradation Language**

When MCP sources are absent, use this phrasing pattern:
"GitHub signals are unavailable - GitHub MCP is not configured. PR and commit data cannot be included."

Do not say "I can't access GitHub" (makes it about the tool). Say what data is absent and why it matters.
