**Front matter (YAML):**
```yaml
---
description: "Generate a sprint retro agenda seeded with sprint facts. Attributed to work items, never individuals."
argument-hint: "[--sprint <sprint-id>]"
allowed-tools: Read, Bash(date +%Y-%m-%d), Bash(date +%Y-W%V), Bash(ls .team-health/pulse-history/)
disable-model-invocation: true
---

Current date: !`date +%Y-%m-%d`
Current ISO week: !`date +%Y-W%V`
```

Note: NO `Write` in allowed-tools — retro-prep makes zero state writes.

**Pre-flight Check:** Copy verbatim from pulse.md.

**Required Reference Reads:** Read SIGNALS.md, PRIVACY.md. (BASELINES.md optional — read if context on signal computation is useful.) Instruction: "Follow these documents exactly."

**Phase A — Sprint Window Determination:**
Parse $ARGUMENTS:
- If "--sprint <id>" present and sources.jira is true: Use Jira MCP to look up that sprint's start and end dates. State: "Retro scope: Sprint [id] ([start] to [end])"
- If "--sprint <id>" present but sources.jira is false: Tell user "Jira MCP not configured — cannot look up sprint [id]. Run without --sprint flag to use pulse history window instead." Stop.
- If no --sprint flag and sources.jira is true: Query Jira MCP for most recently completed sprint. Use that sprint's dates. State: "Retro scope: [sprint name] ([start] to [end])"
- If no --sprint flag and sources.jira is false: Use most recent N pulse history weeks where N = sprint_cadence_weeks from config.json (default 2). State: "Retro scope: last [N] weeks from pulse history (no Jira MCP — sprint boundary unavailable)"

**Phase B — Sprint Data Collection:**

Pulse history (always, if files exist):
1. Run: `Bash(ls .team-health/pulse-history/)`
   - If error or empty: note "No pulse history available." Continue with MCP data only if sources active. If no MCPs and no history: tell user, stop.
2. Filter to files within sprint window
3. Read each qualifying file
4. Extract team-level flag summary: "Week X: N flags" per week
5. Note trend: improving, stable, or degrading signal health across weeks

GitHub queries (if sources.github is true):
- Scope ALL queries to github_org from config.json
- PRs merged in sprint window: count and list titles (NO author names in output)
- PRs with >3 revision cycles (high review round-trips): list titles only
- PR review cycle time distribution: avg, max (NO per-person breakdowns)
- Reverted or hotfixed PRs: list titles

Jira queries (if sources.jira is true):
- Sprint velocity: tickets closed vs. planned
- Carry-over count: tickets started but not closed in sprint
- Blocked tickets at sprint end: count and ticket summaries (NOT assignees)
- Tickets in progress >2 sprints: count

CRITICAL: Do NOT query per-person breakdowns for retro-prep. All queries are team-scoped or work-item-scoped.

**Phase C — Output (retro agenda format):**
Render this exact format:

```

*Data sources: [active MCP sources] and [N] pulse history snapshots*
*Note: All discussion seeds are attributed to work items, not individuals.*

---

## Sprint Facts

**Velocity:** [N tickets closed] / [M planned] — [carry-over count] carried over
**PRs:** [N merged], [M reverted or hotfixed], avg review cycle [X days]
**Blockers resolved:** [N] / [M remaining at sprint end]

[If Jira unavailable: show only GitHub PR data and pulse history. State: "Ticket velocity unavailable — Jira MCP not configured."]
[If GitHub unavailable: show only Jira data and pulse history. State: "PR data unavailable — GitHub MCP not configured."]
[If neither available: show only pulse history summary. State: "Sprint facts limited to pulse history — no GitHub or Jira MCP configured."]

---

*Team discusses. Add your own.*

**[ Team discussion space — what else went well? ]**
```
