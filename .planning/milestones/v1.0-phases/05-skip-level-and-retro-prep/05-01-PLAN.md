Create `.claude/commands/team-health/skip-level.md` with these exact sections in order:

**Front matter (YAML):**
```yaml
---
description: "Generate an upward-facing team brief for your skip-level meeting. People log content excluded by default."
argument-hint: "[--weeks N | --quarter] [--include-person <name>]"
allowed-tools: Read, Write, Bash(date +%Y-%m-%d), Bash(date +%Y-W%V), Bash(ls .team-health/pulse-history/)
disable-model-invocation: true
---

Current date: !`date +%Y-%m-%d`
Current ISO week: !`date +%Y-W%V`
```

**Pre-flight Check:** Copy verbatim from pulse.md (read config.json, check setup_complete, stop if not configured).

**Privacy Gate section (CRITICAL — this is the SKIP-03/SKIP-04 enforcement):**
```
## Privacy Gate

CRITICAL — read this before any data collection:

Do NOT read any files under .team-health/people/ during this command.
People log content is excluded from skip-level output by default.

Exception: if $ARGUMENTS contains "--include-person <name>":
  1. Resolve <name> to slug using config.json team array (case-insensitive prefix match, same as prep.md)
  2. Read .team-health/people/<slug>.json
  3. Include that person's log themes in the People Themes section, labeled as "(opt-in content)"
  4. All other individuals remain unnamed in the output

Default behavior (no --include-person flag):
  - All pulse aggregations use counts only: "N team members", never individual names
  - No people log content in any section
  - This is enforced structurally — skip the Read call entirely for people files
```

**Required Reference Reads:** Same as pulse.md — read SIGNALS.md, BASELINES.md, PRIVACY.md in order. Add instruction: "Follow these documents exactly. Do not improvise signal thresholds, flagging rules, or output language."

**Phase A — Argument Parsing and Lookback Window:**
Parse $ARGUMENTS for flags:
- If "--quarter" present: lookback_start = first day of current calendar quarter
- If "--weeks N" present (N is positive integer): lookback_start = today - (N * 7 days)
- Otherwise (default): read config.json.last_skip_level
  - If last_skip_level is a valid YYYY-MM-DD date: lookback_start = last_skip_level
  - If last_skip_level is null: lookback_start = today - 14 days (2-week default)
- State which source determined the window: "Covering [N] weeks since [date] ([source])"
- Also parse "--include-person <name>" if present (handled by Privacy Gate rules above)

**Phase B — Pulse History Discovery and Loading:**
1. Run: `Bash(ls .team-health/pulse-history/)`
   - If error or empty: note "No pulse history found for this period." Continue with note about limitation.
2. Parse listing — files named YYYY-WNN.md
3. Filter to files where YYYY-WNN falls within [lookback_start, today]
4. Read each qualifying file using Read tool
5. Store contents in memory as PULSE_HISTORY array (oldest to newest)

**Phase C — Aggregation (team-level only by default):**
From pulse history files:
- Count total flagged members per week (extract from "Flagged Members" sections)
- Identify signal types mentioned in flags (PR review lag, commit days, etc.)
- Track flag trend across weeks: improving (fewer flags), stable, or degrading (more flags)
- Extract team-level delivery metrics: aggregate PR counts, ticket counts mentioned in summary sections

CRITICAL aggregation rule: "When reading pulse history files, extract team-level aggregate counts only. Do NOT carry forward individual names from the 'Flagged Members' sections into the brief output. Aggregate as: 'N team members were flagged in week X.'"

If --include-person was passed: also read that person's people log file (per Privacy Gate) and extract log themes (recurring categories, open commitments summary — no raw entry content).

**Phase D — Output (5-section brief):**
Render this exact format:

```
# Team Brief — [lookback period description]

*Covering: [lookback_start] to [today] ([N weeks])*
*Sources: [active pulse history weeks count] pulse snapshots | MCP: [active sources from config]*

---

## 1. Delivery Status

[Team-level velocity from pulse history: PRs merged, tickets closed — aggregated across team, not per person]
[If no pulse history: "Delivery data unavailable — no pulse history for this period."]

---

## 2. Risks and Watch Areas

[Team-level aggregate flags from pulse history]
[Use count language: "N team members showed [signal type] signals during this period — worth monitoring as a group trend."]
[NEVER use individual names here unless --include-person was passed for that specific person]
[If no flags in any week: "No team-level risk signals in this period."]

---

## 3. People Themes

[Patterns from pulse flags — anonymized by default]
[E.g., "Two team members had elevated PR review lag for 2+ consecutive weeks."]
[With --include-person: add labeled subsection for that person's log themes: "### [Name] (opt-in content)" with themes from their people log]
[Without --include-person: no people log content whatsoever]

---

## 4. Asks / Escalations

[ Add escalation items here before sharing this brief ]

*This section is for items requiring your manager's attention. Edit the brief to add your asks before the meeting.*

---

## 5. Team Wins

[Positive delivery signals from pulse history — PRs merged, velocity above baseline, etc.]
[Frame factually: "Team shipped [X] PRs in [week range]"]
[If no positive signals found: "No specific delivery highlights identified in pulse data for this period."]

---

*This brief reflects behavioral data signals. People log content is excluded by default.*
```
