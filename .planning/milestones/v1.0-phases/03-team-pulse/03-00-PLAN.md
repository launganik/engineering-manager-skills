Create `docs/testing/phase-3-smoke-tests.md` with 10 numbered smoke test scenarios. The file must follow this structure exactly:

**Header block:**
- Phase: 3 - Team Pulse
- Command under test: `/team-health:pulse`
- Prerequisites section: Phase 1 setup complete, `.team-health/config.json` exists with `setup_complete: true`, at least two team members configured. Use canonical test fixtures throughout all scenarios:
  - Alice Chen / slug `alice-chen` / github_username: `achen` - baseline data present (8 weeks)
  - Bob Smith / slug `bob-smith` / github_username: `bsmith` - baseline data present (8 weeks), will be the flagged person
  - Carol Davis / slug `carol-davis` / github_username: `cdavis` - NEW team member, no baseline data yet
  - `sources`: `{ "github": true, "jira": false, "slack": false, "calendar": false }` (GitHub-only config for most scenarios)

**Scenario 1 - PULSE-01: Command scans all configured team members**
- Precondition: config.json has 3 team members (Alice, Bob, Carol). `baselines.json` exists for Alice and Bob; Carol has no baseline entry.
- Invocation: `/team-health:pulse`
- Pass criteria: Output mentions all 3 team members (Alice Chen, Bob Smith, Carol Davis) in the summary table. No team member is silently skipped. Carol's row shows "(baseline pending - 0 weeks of data)" instead of a green/yellow/red status.

**Scenario 2 - PULSE-02: Scoring uses personal baseline, not team averages**
- Precondition: Alice has a strong baseline (avg 4 PR reviews/week). Bob has a weaker baseline (avg 1 PR review/week). This week both reviewed 2 PRs.
- Invocation: `/team-health:pulse`
- Pass criteria: Alice is flagged (2 vs. 4 baseline = significant drop). Bob is NOT flagged (2 vs. 1 baseline = above average). Output must reference each person's own baseline avg - NO language comparing one person to another. No phrase like "team average" or "compared to team".

**Scenario 3 - PULSE-03: Correct signals tracked per available source**
- Precondition: `sources.github = true`, all others false.
- Invocation: `/team-health:pulse`
- Pass criteria: Output includes a "Sources active: GitHub" header line. Output explicitly names Jira, Slack, Calendar as unavailable. GitHub signals checked: PR merges per week, PR review count per week, PR review lag (days), commit days per week (4 signals). The 7 signals from Jira/Slack/Calendar are listed as unavailable, not silently absent.

**Scenario 4 - PULSE-04: Conservative two-signal flagging rule**
- This scenario requires two sub-tests, run back-to-back with different baselines.json states:

  Sub-test A (should NOT flag):
  - Setup: Bob has 1 signal at 1.5 stddev below baseline. All other signals within normal range.
  - Pass criteria: Bob shows GREEN in the summary table. No detail section for Bob.

  Sub-test B (should flag YELLOW):
  - Setup: Bob has 2 signals each at ≥1 stddev below baseline (e.g. PR review count −1.5 stddev AND commit days −1.2 stddev).
  - Pass criteria: Bob shows YELLOW in the summary table. A detail section appears for Bob listing both triggering signals with their deltas.

  Sub-test C (should flag RED):
  - Setup: Bob has 1 signal at ≥2 stddev below baseline (e.g. PR review count: 0 reviews vs. baseline 4.2/week = −4.2, −6 stddev).
  - Pass criteria: Bob shows RED in the summary table. Detail section present with that single signal.

**Scenario 5 - PULSE-05: Summary table includes all members; detail sections for flagged only**
- Precondition: Alice is GREEN, Bob is YELLOW, Carol is baseline-pending.
- Invocation: `/team-health:pulse`
- Pass criteria:
  1. Summary table has exactly 3 rows - one per team member.
  2. Bob has a detail section below the table.
  3. Alice does NOT have a detail section (GREEN = table row only).
  4. Carol does NOT have a detail section (pending = table row only with pending label).

**Scenario 6 - PULSE-06: Flagged signal output states metric, delta, and source**
- Precondition: Bob is YELLOW, flagged on PR review count (0 reviews vs. baseline 3.8/week).
- Invocation: `/team-health:pulse`
- Pass criteria: Bob's detail section contains ALL of:
  - Metric name: "PR review count"
  - Current value: "0 reviews this week"
  - Baseline avg: "8-week avg 3.8/week"
  - Delta: "−3.8" (absolute) AND "−5.4 stddev" (standard deviation units)
  - Source: "GitHub MCP"
  - Must NOT say "engagement indicators" or any opaque description

**Scenario 7 - PULSE-07: Disclaimer appears verbatim at end of every output**
- Precondition: Any valid pulse run (use Scenario 1 state).
- Invocation: `/team-health:pulse`
- Pass criteria: The last paragraph of the pulse output is EXACTLY:
  "These signals are indicators, not diagnoses. Always talk to your people before drawing conclusions."
  No paraphrase accepted. Must appear even when all team members are GREEN.

**Scenario 8 - PULSE-08: baselines.json written with provenance fields after each run**
- Precondition: `baselines.json` exists with Alice's baseline data.
- Invocation: `/team-health:pulse`
- Pass criteria: After the run, open `.team-health/baselines.json` and verify:
  - Top-level: `schema_version: "1"`, `last_updated: [today's date]`
  - Per person (Alice at minimum): `computed_from`, `computed_to`, `source` (array), `window_weeks`
  - Per metric: `window` (array), `mean`, `stddev`, `last_value`
  - `computed_to` equals today's run date
  - Carol's entry is created with `window: [<current week value>]`, `mean = current_value`, `stddev = 0` (first-run behavior)

**Scenario 9 - PULSE-09: Pulse history snapshot written to pulse-history/**
- Precondition: `.team-health/pulse-history/` does NOT exist (fresh install).
- Invocation: `/team-health:pulse`
- Pass criteria:
  - Directory `.team-health/pulse-history/` is created.
  - File `.team-health/pulse-history/<YYYY-WNN>.md` exists (where YYYY-WNN is the current ISO week, e.g. `2026-W11`).
  - File contains: run date, sources active/unavailable, summary table (all members), flagged member detail, disclaimer.
  - Running pulse again the following week creates a NEW file (not overwriting the previous one).

**Scenario 10 - PULSE-10: Graceful degradation names all unavailable sources**
- Precondition: Set config.json `sources` to `{ "github": false, "jira": false, "slack": false, "calendar": false }`.
- Invocation: `/team-health:pulse`
- Pass criteria:
  - Output does NOT error or crash.
  - Output explicitly names all 4 sources as unavailable: GitHub, Jira, Slack, Calendar.
  - Each unavailable source uses the degradation phrasing from PRIVACY.md (not invented phrasing).
  - Summary table still renders with "baseline pending" or "no signals available" for each person.
  - Disclaimer still appears at the end.
